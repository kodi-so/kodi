# OpenClaw Bridge — Infrastructure & Env Reference

This file documents the env vars, AWS resources, and external integrations the
`kodi-bridge` plugin and its Kodi-side counterparts depend on. It's the
counterpart to `architecture-plan.md` for ops / deploy.

## Composio (Kodi side, KOD-388)

The Kodi API uses one master Composio API key + one webhook subscription per
environment. Per-user identity is just the Kodi `user_id` passed as the
Composio `userId` parameter; we don't issue per-user API keys.

### Required

- `COMPOSIO_API_KEY` — master API key. Used for `tools.list`, action
  execution, OAuth-config provisioning, webhook verification. Without it
  the agent loadout builder degrades to `composio_status: 'failed'` and
  no tools are exposed to the agent.

### OAuth + webhooks

- `COMPOSIO_BASE_URL` — overrides the default `https://backend.composio.dev`
  (only set in non-prod or when sharding is in play).
- `COMPOSIO_OAUTH_REDIRECT_URL` / `COMPOSIO_AUTH_CALLBACK_URL` — the URL
  Composio redirects users back to after OAuth. Falls back to
  `${APP_URL}/integrations` when neither is set.
- `COMPOSIO_WEBHOOK_SECRET` — signing secret for inbound Composio
  webhooks at `/integrations/composio/webhook`. KOD-386's rotation
  triggers fire from this handler.

### Per-toolkit auth configs

These hold the Composio-side auth-config IDs Kodi created for our own OAuth
apps. Optional — toolkits without a configured ID fall back to Composio-managed
auth where available. If a toolkit needs custom auth and no ID is set, the
`createConnectLink` mutation surfaces a clear error.

- `COMPOSIO_AUTH_CONFIG_GOOGLE`
- `COMPOSIO_AUTH_CONFIG_SLACK`
- `COMPOSIO_AUTH_CONFIG_GITHUB`
- `COMPOSIO_AUTH_CONFIG_LINEAR`
- `COMPOSIO_AUTH_CONFIG_NOTION`

### Session defaults

- `COMPOSIO_SESSION_DEFAULT_TOOLKITS` — comma-separated toolkit slugs.
  When an org has no `toolkit_policies` rows yet, the agent loadout
  builder uses this list as the default allowlist (intersected with
  whatever the user has actually connected). Once an admin saves any
  policy, the org-level table takes over and this default is ignored.

  Empty / unset preserves the legacy "every active connection is enabled"
  behavior, so existing environments don't change behavior unless the
  variable is explicitly set.

  Example: `COMPOSIO_SESSION_DEFAULT_TOOLKITS=gmail,slack,googlecalendar`

### Behavior matrix

| Org has policy rows | Env default set | Effective allowlist |
|---|---|---|
| Yes | (any) | ACTIVE connections ∩ policies-where-`enabled=true` |
| No | Yes | ACTIVE connections ∩ env default |
| No | No / empty | ACTIVE connections (everything) |

## Plugin bundle distribution

### Env vars

- `PLUGIN_BUNDLE_S3_BUCKET` — private S3 bucket holding `kodi-bridge` plugin
  bundles. Object key convention: `bundles/<version>/kodi-bridge.tgz`.
- `PLUGIN_BUNDLE_S3_REGION` — AWS region for the bucket.
- `PLUGIN_BUNDLE_URL_TTL_SECONDS` — TTL for signed download URLs handed
  to instances. Default 600.

### Bucket provisioning (KOD-357)

One bucket per environment, in the existing Kodi AWS account. Buckets are
private; instances never hit S3 directly — the Kodi API generates a
short-lived signed URL (default TTL 600s) that the plugin's self-update
module downloads through.

| Env | Bucket name | Region |
|---|---|---|
| dev | `kodi-plugin-bundles-dev` | `us-east-1` |
| staging | `kodi-plugin-bundles-staging` | `us-east-1` |
| prod | `kodi-plugin-bundles-prod` | `us-east-1` |

Region choice: bundles are small (< 10 MB) and downloaded once per
instance per release, so cross-region latency is negligible. We pin
`us-east-1` to match the rest of the Kodi AWS footprint.

#### Required bucket settings

- **Block all public access**: every option ON (acls + bucket policy +
  ignore_public_acls + restrict_public_buckets). The bucket is reachable
  only via signed URLs generated by Kodi.
- **Versioning**: enabled. Republishing the same `<version>` is rejected
  by the publish endpoint (see `apps/api/src/routes/plugin-versions.ts`),
  but versioning gives us recovery if a bundle is accidentally overwritten
  or deleted.
- **Default encryption**: SSE-S3 (`AES256`). KMS isn't required —
  bundles are public-domain code distribution, not secrets.

#### Manual provisioning via AWS CLI

The Kodi monorepo doesn't Terraform-manage storage today; provisioning is
manual but reproducible from the snippet below. Run it once per env with
the matching `--region` and `--profile`. Idempotent — safe to re-run.

```sh
BUCKET=kodi-plugin-bundles-dev   # -staging / -prod for the other envs
REGION=us-east-1
PROFILE=kodi-dev                 # the AWS-CLI profile for that env

# 1. Create the bucket. us-east-1 doesn't take a LocationConstraint; other
#    regions do. Adjust if you ever change REGION.
aws s3api create-bucket \
  --bucket "$BUCKET" \
  --region "$REGION" \
  --profile "$PROFILE"

# 2. Block all public access (every flag ON).
aws s3api put-public-access-block \
  --bucket "$BUCKET" \
  --public-access-block-configuration \
    'BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true' \
  --profile "$PROFILE"

# 3. Enable versioning.
aws s3api put-bucket-versioning \
  --bucket "$BUCKET" \
  --versioning-configuration Status=Enabled \
  --profile "$PROFILE"

# 4. Default encryption (SSE-S3).
aws s3api put-bucket-encryption \
  --bucket "$BUCKET" \
  --server-side-encryption-configuration \
    '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"},"BucketKeyEnabled":true}]}' \
  --profile "$PROFILE"
```

Verify: `aws s3api get-bucket-versioning --bucket "$BUCKET"` should show
`Status: Enabled`; `get-public-access-block` should show every value
`true`; `get-bucket-encryption` should show the SSE-S3 rule.

#### IAM policy for the Kodi API service account

The API needs `PutObject` (publish endpoint, `apps/api/src/routes/plugin-versions.ts`),
`GetObject` (signed URLs handed to instances), and `HeadObject` (existence
check before issuing a URL). Scope to `bundles/*` so an over-broad
credential leak can't read or overwrite anything outside the bundle prefix.

```jsonc
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "KodiApiPluginBundlesRW",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:HeadObject"
      ],
      "Resource": "arn:aws:s3:::kodi-plugin-bundles-<env>/bundles/*"
    },
    {
      "Sid": "KodiApiPluginBundlesList",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::kodi-plugin-bundles-<env>",
      "Condition": {
        "StringLike": { "s3:prefix": ["bundles/*"] }
      }
    }
  ]
}
```

`ListBucket` (prefix-scoped) is included for ops/debug — the runtime
doesn't need it, but it's the difference between "bundle missing → 404
when ops greps" and "bundle missing → AccessDenied with no diagnostic".
The `s3:prefix` condition keeps the listing scope to the same `bundles/*`
prefix the runtime is allowed to touch.

The IAM principal is whichever identity the Kodi API runs under in each
environment (typically a Railway service token bound to an AWS access key
pair stored in `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`, or — once
we move to ECS/EKS — an instance role). The `S3Client` in
`apps/api/src/lib/plugin-bundles.ts` uses the standard AWS credential
chain (env vars > shared config > IAM role), so attach the policy to
whichever surface is active for the env.

Attach the policy:

```sh
# Inline against an IAM user (Railway-style env-var creds):
aws iam put-user-policy \
  --user-name kodi-api-dev \
  --policy-name KodiPluginBundlesRW \
  --policy-document file://kodi-plugin-bundles-policy.json \
  --profile "$PROFILE"

# Or attached to a managed policy / role for ECS-style deploys.
```

#### Bucket ARNs

above and (eventually) by Terraform if/when we adopt it.

| Env | ARN |
|---|---|
| dev | `arn:aws:s3:::kodi-plugin-bundles-dev` |

#### When we eventually Terraform this

If/when we Terraform-manage storage, the equivalent module is roughly:

```hcl
resource "aws_s3_bucket" "plugin_bundles" {
  bucket = "kodi-plugin-bundles-${var.env}"
}

resource "aws_s3_bucket_public_access_block" "plugin_bundles" {
  bucket                  = aws_s3_bucket.plugin_bundles.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_versioning" "plugin_bundles" {
  bucket = aws_s3_bucket.plugin_bundles.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "plugin_bundles" {
  bucket = aws_s3_bucket.plugin_bundles.id
  rule {
    apply_server_side_encryption_by_default { sse_algorithm = "AES256" }
    bucket_key_enabled = true
  }
}
```

## Plugin-instance HMAC

Per-instance shared secret used to sign:

- Outbound (plugin → Kodi): event ingest at `/api/openclaw/events`,
  reconcile fetch at `/api/openclaw/agents`, etc.
- Inbound (Kodi → plugin): admin reload, agent provision/deprovision,
  subscription updates.

The secret is generated at instance provisioning time, stored encrypted
in `instances.plugin_hmac_secret_encrypted`, and baked into cloud-init.
Rotation lives in KOD-412 (M8) and is out of scope here.
